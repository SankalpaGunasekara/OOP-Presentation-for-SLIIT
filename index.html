Understanding OOP

The 3 Big Questions

1. WHAT?

Real World Modeling

It is a coding style where we organize software based on Objects (Things), not just Actions.


Ex: Instead of "Variables", we build "Cars", "Students", "Banks".

2. WHY?

To Fix "Spaghetti Code"

The Old Way (Procedural): Data was loose and messy. Hard to manage.

The OOP Way: Code is modular, clean, and reusable.

3. HOW?

Bundling (Encapsulation)

We bundle the Data and the Logic together into one unit.


Ex: A "Car Object" holds its own fuel level AND its own drive function.

01. Class vs. Object

WHAT (Concept)

Class: The Blueprint. It defines the rules.

Object: The actual thing created from that blueprint.

WHY (Benefit)

You design the car once (Class), but you can manufacture thousands of them (Objects) without redrawing the plans. 

// 1. The Blueprint (Logic)
class Car {
    String color;
    String engineType;
}

// 2. The Real World Item (Memory)
Car myAudi = new Car();
Car myToyota = new Car();

02. Encapsulation

WHAT

Wrapping data inside a protective shield.

You hide the messy engine wires and only expose the steering wheel.

WHY

Security. It prevents other parts of the code from accidentally breaking the internal data. 

class Car {
    // Private: No one can touch this directly
    private int speed;

    // Public: The controlled way to change speed
    public void accelerate(int amount) {
        if (amount > 0) {
            speed = speed + amount;
        }
    }
}

03. Abstraction

WHAT

Focusing on what something does, not how it works.

You drive using a pedal, ignoring the fuel injection physics.

WHY

Simplicity. It hides complex logic so the developer can focus on high-level architecture. 

// The User Interface
interface Drivable {
    void drive(); 
    // We don't care HOW it drives yet
}

// The Implementation
class Tesla implements Drivable {
    public void drive() {
        // Complex electric motor logic...
    }
}

04. Inheritance

WHAT

A child class automatically gets the fields and methods of a parent class.

WHY

Reusability. A 'SportsCar' is just a 'Vehicle' plus a turbo. Don't rewrite the wheel logic! 

// The Base Class
class Vehicle {
    int wheels = 4;
}

// The Child Class
class SportsCar extends Vehicle {
    boolean hasTurbo = true;
    
    // SportsCar now has BOTH 
    // 'wheels' and 'hasTurbo'
}

05. Polymorphism

WHAT

"Many Forms". One method name behaves differently based on the object type.

WHY

Flexibility. You can press the "Gas Pedal" on any car, and it will go, without needing to know if it's a Truck or a Ferrari. 

class Truck extends Vehicle {
    void go() { print("Heavy Start..."); }
}

class Ferrari extends Vehicle {
    void go() { print("Zoom!"); }
}

// The Magic:
Vehicle v = new Ferrari();
v.go(); // Outputs: "Zoom!"
